# Как запустить и автоматизировать миграцию данных BSRegister: От хаоса ID к чистому плану

Вы сталкивались с проблемой устаревших идентификаторов? В больших системах данные из разных источников — Яндекс Диск, старые базы — накапливают «архив» ID, который невозможно просто прочитать. Этот хаос ID (`BS-<КодРегиона>-<УникальныйНомер>`) делает невозможным автоматическое управление контентом и вычисления правильного нового имени.

Мы создали **модульную, гибридную систему аудита**, которая решает эту проблему, превращая сырые метаданные в чистый, *неисполняемый* **План Миграции**.

Эта статья не просто объясняет, как работает наш код. Это путеводитель, который покажет вам, как вы можете использовать эту систему, чтобы добиться максимальной чистоты и структуры в вашей базе данных.

---

## 💡 Что делает система? (The Core Concept)

Вместо того, чтобы пытаться написать сложный, многоэтапный скрипт, который будет работать с десятком исключений, мы создали единый, централизованный конвейер (Pipeline).

**Наша цель:** Взять сырой каталог файлов и получить один файл: `reports/migration_plan_[timestamp].json`.

Этот отчет — это **аудит**. Он говорит: "Вот эти файлы были такими, они должны стать такими, и вот почему". Он не командует удалением или изменением; он только консультирует.

### 🔍 Как мы управляем хаосом ID?

Основная сложность — в унификации старых идентификаторов БС. Мы внедрили двухступенчатую логику:

1.  **Парсинг Regex (BSID):** Модуль `bsRegexParser.js` обрабатывает сырые строки и стандартизирует их до унифицированного вида `BS-<КодРегиона>-<УникальныйНомер>`. Это повышает надежность на 95% для всех структурных ID.
2.  **Гибридная Классификация:** Для всего остального — категории и нового названия — используется **гибридный подход**:
    *   **Логический парсинг (Regex):** Проверка на ключевые слова или известную структуру.
    *   **AI-Классификация (Ollama/Gemini):** Когда правила недостаточны, задействуется локальная LLM. Мы отправляем LLM метаданные, просим определить категорию и предложить лучшее имя. Это позволяет выявить неструктурированные связи, которые невозможно поймать регулярным выражением.

## 🛠️ Как это использовать? (The Workflow)

Работа с системой миграции происходит в три последовательных шага, которые вы инициируете через единую команду в CLI:

1.  **Инициализация:** Запуск процесса с нужной папки.
2.  **Обнаружение (Discovery):** Система собирает сырые метаданные и ID из указанной директории.
3.  **Генерация Плана:** Ядро (`migration-engine.js`) запускает полную гибридную цепочку обработки и записывает результат.

### 🚀 Пример рабочего процесса:

Для начала аудита вам нужно выполнить одну команду:

`node index.js --compile-migration-plan <путь_к_директории>`

Что происходит под капотом?

*   `index.js` (Оркестратор) вызывает `migrationEngine.runPlan()`.
*   Ядро получает список файлов и начинает итерацию.
*   Каждый файл проходит через цикл: `bsRegexParser` -> `file-system` (получение метаданных) -> `ollamaClient` (классификация).
*   Результат каждой итерации формирует объект `MigrationFile`, который затем агрегируется в единый `MigrationReport`.

### ✨ Что вы получите в итоге?

В папке `reports/` вы найдете файл `migration_plan_[timestamp].json`.

**Обратите внимание:** Этот файл — **только чтение**. Он содержит структуру данных и рекомендации. Он никогда не запускает удаление, не меняет имена и не копирует файлы физически.

**Пример структуры JSON (Аудит):**
```json
{
  "report_date": "2026-05-25T09:14:28Z",
  "processed_files_count": 452,
  "migration_records": [
    {
      "original_id": "BS-12-A12345",
      "original_name": "Документ_старый.pdf",
      "suggested_category": "Архивные отчеты",
      "suggested_new_name": "Report_Archive_2023_12_Compliance",
      "action_required": "Review_Manual"
    }
    // ... другие записи
  ]
}
```

## 🔮 Что дальше? (Inspiration Section)

Эта система — лишь первый шаг к полной цифровизации. Получив чистый, верифицированный план миграции, вы можете:

*   **Автоматизировать следующие шаги:** Используйте отчет JSON как источник правды для написания скриптов-обработчиков.
*   **Сравнивать версии:** Запустите этот же процесс на основе изменений из другого источника (например, из Google Drive), и сравните два отчета для выявления расхождений в классификации.
*   **Обучить AI:** Используйте сгенерированный `MigrationReport` как набор идеальных примеров для дальнейшего обучения вашего корпоративного ИИ.
